    Introduction

    Hey Friends, in this post we are going to learn below AWS Lambda related topics,

    1. What is serverless
    2. What is AWS Lambda
    3. How to run Go code in AWS Lambda.

    Serverless

    So what is serverless ?, as per Wikipedia,

    Serverless computing is a cloud computing execution model in which the cloud provider allocates machine resources on demand, taking care of the servers on behalf of their customers. “Serverless” is a misnomer in the sense that servers are still used by cloud service providers to execute code for developers.

    Ok so basically, we have some set of code to run but we don’t want to manage the server, scaling, resources, etc stuff so we can use serverless instead. We will give the code, and specify when the code should run. We will be charged based on usage.

    AWS Lambda

    Let’s see what AWS themselves has to say about Lambda,

    AWS Lambda is a serverless compute service that runs your code in response to events and automatically manages the underlying compute resources for you.

    So lambda is the service we will use to deploy our code. We can then invoke our Lamda with AWS CLI or through various other triggers provided by Lambda. Lamba invocations are stateless.

    Our Code

    Our main function code will be something like the below image, start will accept objects with the below rules,

    * handler must be a function
    * handler may take between 0 and two arguments.
    * if there are two arguments, the first argument must satisfy the “context.Context” interface.
    * handler may return between 0 and two arguments.
    * if there are two return values, the second argument must be an error.
    * if there is one return value it must be an error.

    Also, the valid handler signatures are, wherein TIn and TOut are JSON package supported objects.

    func ()
    func () error
    func (TIn) error
    func () (TOut, error)
    func (TIn) (TOut, error)
    func (context.Context) error
    func (context.Context, TIn) error
    func (context.Context) (TOut, error)
    func (context.Context, TIn) (TOut, error)

    package main

import (
	"context"

	"github.com/aws/aws-lambda-go/lambda"
)

type Request struct {
	Email string `json:"email"`
	Name  string `json:"name"`
	Age   string `json:"age"`
	City  string `json:"city"`
}

type Response struct {
	Status  string `json:"status"`
	Message string `json:"message"`
}

var userCache = map[string]Request{}

func HandleRequest(ctx context.Context, req Request) (Response, error) {

	if req.Email == "" || req.Name == "" || req.Age == "" || req.City == "" {
		return Response{Status: "error", Message: "all fields are required"}, nil
	}

	if isCached(req.Email) {
		return Response{
			Status:  "success",
			Message: "User already present in cache of this container",
		}, nil
	}

	//cache the user
	userCache[req.Email] = req

	return Response{
		Status:  "success",
		Message: "User cached",
	}, nil
}

func isCached(email string) bool {
	if _, ok := userCache[email]; ok {
		return true
	}
	return false
}

func main() {
	lambda.Start(HandleRequest)
}

    If you see the code, we can see that I am using a map to cache the data, but I said that the lambda is stateless before so how will this cache work? Well, the lambda functions are executed in ec2 containers themselves so if the same container is allocated for our invocation then we can access the cached data.

    To cache an object we have to define it as a Global Variable, in our case userCache is a global variable.

    Build, Zip and Upload

    Let’s build our binary now and upload it to our Lambda function.

    Our build command is – go build -o hello main.go, Our binary name will be hello because earlier we saw the handler mentioned was hello. We can also change the handler name in Lambda and change the binary name accordingly.

    We can’t directly upload it. We need to zip it and then upload it.

    Our zip command is – zip hello.zip hello, this will create hello.zip file and add our binary to it.
