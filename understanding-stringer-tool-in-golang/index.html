    Introduction

    Hi guys, let’s take a look at the stringer command or tool in this post.

    Godoc defines the stringer tool as,

    Stringer is a tool to automate the creation of methods that satisfy the fmt.Stringer interface. Given the name of a (signed or unsigned) integer type T that has constants defined, stringer will create a new self-contained Go source file implementing.

    ok, so the stringer command basically generates string methods that are satisfied by the fmt.Stringer interface. So it will create the string methods for us instead of creating them manually. This is an alternative for the missing enum type in Golang.

    Use Case

    We first declare a custom type,

    type PhoneModel int

    Consider we are writing some code that needs constants for iPhones. 

    const (
        iPhone4  PhoneModel = iota //0
        iPhone5                    //1
        iPhone6                    //2
        iPhone7                    //3
        iPhone8                    //4
        iPhone10                   //5
        iPhone11                   //6
        iPhone12                   //7
        iPhone13                   //8
    )

    We are using iota to avoid giving constants the numbers manually. Let’s say our application is printing some error and we have a logline as below,

    fmt.Printf(“Some Error occurred when listing phone : %v , error : %v\n”, iPhone11, “Some error”)

    The output is as below, 

    Some Error occurred when listing phone : 6 , error : Some error

    If we are checking logs we won’t find out for which phone this error occurred as we are not going to remember the constant numbers. To solve this we can write a String method that satisfies the fmt.Stringer interface. Let’s see how it will look like,

    func (pm PhoneModel) String() string {

        switch pm {
        case 0:
            return “iPhone4”
        case 1:
            return “iPhone5”
        case 2:
            return “iPhone6”
        case 3:
            return “iPhone7”
        case 4:
            return “iPhone8”
        case 5:
            return “iPhone9”
        case 6:
            return “iPhone10”
        case 7:
            return “iPhone11”
        case 8:
            return “iPhone12”
        case 9:
            return “iPhone13”
        }

        return “”
    }

    Let’s try to run our print function again and see what’s the output,

    Some Error occurred when listing phone : iPhone10 , error : Some error

    There is something wrong with this, We were printing iPhone11 here so why is it printing the iPhone10? Well it looks like we made some mistake, iPhone9 does not exist but we still have a case for it. This is a human mistake and could happen.

    Stringer Tool to the rescue

    To avoid this we use the stringer tool which generates the string equivalent of the constants. We use the below command to generate the Stringmethod.

    stringer -type PhoneModel

    our main file is tool.go and our new code created by stringer is in phonemodel_string.go which looks like below.

    I have commented custom Stringmethod now. If we run our code we get the below output,

    akshay@AKSHAY-PC:~/learning-go/stringer/tool$ go run *.go
    Some Error occurred when listing phone : iPhone11 , error : Some error

    As expected iPhone11 is printed now. If we, later on, add new models we just need to add them in the constants and generate the String method for them.

    Conclusion

    To conclude, We have seen how the stringer tool can help us avoid human errors and generate string method for convenience.

    We can also use go:generate to do this instead of running the stringer tool every time. We will see it in another post.
