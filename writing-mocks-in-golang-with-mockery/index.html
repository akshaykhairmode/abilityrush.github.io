    Introduction

    Hello Peeps, today we are going to see how we can write mocks in unit tests using the mockery tool. So what is mockery? mockery gives us the ability to mock the data for the methods which we are calling. This is achieved via interfaces.

    Its recommended to check out the interfaces post to understand this better. We are going to create RedisQueue which will have pop and push methods. We are going to test these methods via mocking.

    We are going to use redigo client for Redis in this tutorial.

    Writing Our Interface

    Redigo provides a Do method on their connection where we can call any of the Redis commands. We will write an interface that will have the same method.

    type Queue interface {
	Do(string, ...interface{}) (interface{}, error)
}

    Mockery will use this interface when generating mocks for it.

    Actual Code

    Let’s go to our actual code. Check out the inline comments.

    package main

import (
	"log"

	"github.com/gomodule/redigo/redis"
)

type Queue interface { //Mockery will use this interface to generate mocks
	Do(string, ...interface{}) (interface{}, error)
}

type RedisQueue struct { //The struct which we will use that is an wrapper of redis connection.
	conn Queue
	name string
}

func (q RedisQueue) Pop() (string, error) { //Our pop method. Does a left pop.

	var data string
	var err error

	data, err = redis.String(q.conn.Do("LPOP", q.name))

	return data, err
}

func (q RedisQueue) Push(data string) error { //Our push method. Does a right push.

	_, err := q.conn.Do("RPUSH", q.name, data)
	return err
}

func main() {

	conn, err := redis.Dial("tcp", "localhost:6379") //Creating connection.
	if err != nil {
		log.Fatal(err)
	}

	queue := RedisQueue{
		conn: conn,
		name: "my-test-queue",
	}

	log.Println(queue.Push("test-data"))
	log.Println(queue.Pop())

}

    The output of this code is as below. When pushing we get a nil error and while popping we get our pushed data.

    Note: I have Redis running on my local machine but its not required when running tests which we will come to later.

    Generating Mocks

    First, we want to generate the mocks.

    We can install mockery with this command – go install github.com/vektra/mockery/v2@latest

    Mockery is installed in my home bin folder so I will run mockery with this command – ~/go/bin/mockery --all 

    Mockery creates a new folder mocks where the autogenerated files are stored. We can see the autogenerated file below. We can use this mocked queue struct in our test.

    Writing Tests Using the Mocks

    Let’s write our first test for Push. First, we need to import the mocks. Our import statement is as below,

    import (
	"fmt"
	"testing"

	"learning-mockery/mocks" //our module name is learning-mockery

	"github.com/stretchr/testify/assert"
)

    Lets move ahead to the test, see the inline comments

    func TestRedisPush_Success(t *testing.T) {

	redisConn := new(mocks.Queue) //create a new object of the mockery queue.

	q := RedisQueue{
		conn: redisConn,
		name: "redis-queue",
	}

	pushData := "some-data"

	//Mockery provides on and return methods.
	//The On method - In the arguments we first pass the function name which we are mocking and then the other parameters. In this case Do is the function which we are calling and other arguments are the ones which are passed when that function is called.
	//The Return method - Here we pass what we expect the Do function to return.
	redisConn.On("Do", "RPUSH", q.name, pushData).Return(nil, nil)
	err := q.Push(pushData)
	assert.NoError(t, err, "Unexpected Error")
}

    If we run the test we get below output. It passes as expected.

    Let’s check our coverage. It’s 50 percent as we have not tested our main function. It looks like below.

    Conclusion

    We see that we can mock the tests easily with mockery. I have added more tests that you guys can see on github.

    The point to note is even though we have a Redis server running on our local server it’s not required as we are mocking the output of the Redis Do method.

    If you want to know what happens when the actual method changes. You can clone from github and modify the methods. The tests should return an error. For example, If we change LPOP to RPOP our tests will fail, this is because mockery will also verify what arguments are passed.
