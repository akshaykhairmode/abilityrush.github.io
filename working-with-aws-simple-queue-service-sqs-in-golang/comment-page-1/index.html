    Introduction

    Hey Folks, today we are going to use AWS Simple Queue Service (SQS) for doing some push and pop operations. We are also going to see how the persistence nature of this queue works.

    First things first, You need an AWS account for this. You can create one account in AWS free tier.

    Getting ID and Secret for SQS

    Assuming your account is created, We will go ahead and get the credentials to connect to SQS from our local machine. We need AWS ID and Secret for that. After login, we go to the options at the top right and click on security credentials.

    On the credentials page, we will see the collapse as below, We need to click on create a new access key. It will generate the ID and Secret. Note it down somewhere or download the file.

    Once we have got the keys, We will set them to environment variables. Click here to see how it’s done.

    We also need the region, we will use export AWS_REGION=ap-south-1. This is the region in which I will create the queues. You have to use the region where you will create the queue.

    SQS Queue Types

    There are 2 types of queues available for us,

    1. Standard – At-least-once delivery, message ordering isn’t preserved (Chances of duplicates)
    2. FIFO – First-in-first-out delivery, message ordering is preserved (No chance of duplicates)

    Other than the above difference mentioned, the standard queue is very fast as compared to FIFO. As duplicates do not always happen it doesn’t matter which queue we use, we will proceed with the standard queue.

    Let’s go ahead and create a queue. We will use the default options. Our queue name will be SQS_TEST_QUEUE.

    Our Code

    For doing the SQS operations, We first need to create an SQS object which will provide us with methods to push and pull messages. This SQS object will also provide us with a method to get the queue url.

    We need the queue url when doing the queue operations. We cant pass the queue name at that time.

    Creating the SQS Object and Getting the Queue URL

    func createSQSObj() (*sqs.SQS, *sqs.GetQueueUrlOutput) {
	sess := session.Must(session.NewSessionWithOptions(session.Options{
		SharedConfigState: session.SharedConfigEnable,
	}))

	svc := sqs.New(sess)
	queueName := "SQS_TEST_QUEUE"

	qurlOut, err := svc.GetQueueUrl(&sqs.GetQueueUrlInput{
		QueueName: &queueName,
	})

	if err != nil {
		log.Fatalln(err)
	}

	return svc, qurlOut
}

    We are creating a session without passing any ID or Secret. They will be picked from our ENV variables. After creating the SQS obj we get the queue url.

    Pushing message to queue

    func pushToQueue(svc *sqs.SQS, qurl *string) {

	now := time.Now().Local().String()
	msgBody := "Hi, I am entered the queue at " + now

	_, err := svc.SendMessage(&sqs.SendMessageInput{
		MessageBody: &msgBody,
		QueueUrl:    qurl,
	})

	if err != nil {
		log.Fatalln(err)
	}

	log.Printf("Pushed to queue, Data : %v", msgBody)
}

    We are using the SQS obj to simply do a send message, We are passing the queue url here.

    Reading from queue

    func readFromQueue(svc *sqs.SQS, qurl *string) {

	var maxNumMsg int64 = 1

	rmo, err := svc.ReceiveMessage(&sqs.ReceiveMessageInput{
		QueueUrl:            qurl,
		MaxNumberOfMessages: &maxNumMsg,
	})

	if err != nil {
		log.Fatalln(err)
	}

	if len(rmo.Messages) <= 0 {
		log.Println("No Messages to read.")
	}

	for _, msg := range rmo.Messages {
		log.Printf("Read from queue, Data : %v", *msg.Body)

		if shouldDelete {
			_, err := svc.DeleteMessage(&sqs.DeleteMessageInput{
				QueueUrl:      qurl,
				ReceiptHandle: msg.ReceiptHandle,
			})

			if err != nil {
				log.Fatalln(err)
			}

			log.Printf("Delete from queue success")
		}

	}

}

    We have the ability to tell SQS how many messages we want to read together. MaxNumberOfMessages defines that. It’s also possible that there are no more messages, in that case, we say no messages available. Then we read from the messages, We will come to the delete part later.

    Our Main Function

    func main() {

	flag.BoolVar(&shouldDelete, "delete", false, "delete after reading")
	flag.BoolVar(&push, "push", false, "push the message")
	flag.BoolVar(&pop, "pop", false, "pop message")
	flag.Parse()

	if !push && !pop {
		log.Fatalln("At least push or pop flag is required.")
	}

	svc, qurlOut := createSQSObj()

	if push {
		pushToQueue(svc, qurlOut.QueueUrl)
	}

	if pop {
		readFromQueue(svc, qurlOut.QueueUrl)
	}

}

    Running our program

    We have added the required code, let’s try testing it.

    Push and Pop Once,

    Only Pop

    If we see here, we are getting the data we pushed again and again. This is because of the persistent nature of the SQS. If we read from something we need to delete it so that it is not read by other consumers.

    We have VisibilityTimeout configuration which can be set for the queue or be set while sending messages. This tells SQS that once one consumer has read the message wait for x amount of time before showing it to other consumers. In this x time we have to delete the message or it’s going to be visible again.

    Pop with Delete

    Once it has been deleted, we cant consume it again. 

    Conclusion

    We have seen how we can create credentials for AWS, use them to connect to SQS from a local machine. How to send a message to the queue and read the message from the queue.
